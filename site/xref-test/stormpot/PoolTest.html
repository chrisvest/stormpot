37  /**
38   * This is the generic test for Pool implementations. The test ensures that
39   * an implementation adheres to the general contract of the Pool interface,
40   * given certain circumstances and standardised configurations.
41   * <p>
42   * Pools may have other properties, and may be configurable to deviate from
43   * the standardised behaviour. However, such properties must not be
44   * observable within the standardised spectrum of configurations.
45   * <p>
46   * The tests for any special properties that a pool may have, must be put in
47   * a pool-specific test case. Do not use assumptions or other tricks to
48   * pollute this test case with tests for pool-specific or non-standard
49   * behaviours and configurations.
50   * <p>
51   * The test case uses theories to apply to the set of possible Pool
52   * implementations. Each implementation must have a PoolFixture, which is
53   * used to construct and initialise the pool, based on a {@link Config}.
54   * <p>
55   * The only assumptions used in this test, is whether the Pool is a
56   * LifecycledPool or not. And most interesting pools are life-cycled.
57   * LifecycledPools can be shut down. This is a required ability, in order to
58   * test a number of behaviours, but also brings about its own set of new
59   * behaviours and flows that needs to be tested for. Those tests are also
60   * included in this test case.
61   * <p>
62   * <strong>Note:</strong> when adding, removing or modifying tests, also
63   * remember to update the {@link Pool} javadoc - especially the part about
64   * what promises are provided by the Pool interface and its implementations.
65   * 
66   * @author Chris Vest &lt;mr.chrisvest@gmail.com&gt;
67   * @see Pool
68   */

347   /**
348    * Objects in the pool only live for a certain amount of time, and then
349    * they must be replaced/renewed. Pools should generally try to renew
350    * before the timeout elapses for the given object, but we don't test for
351    * that here.
352    * We set the TTL to be 1 milliseconds, because that is short enough that
353    * we can wait for it in a spin-loop. This way, the objects will always
354    * appear to have expired when checked. This means that every claim will
355    * always allocate a new object, and so our two claims will translate to
356    * at least two allocations, which is what we check for. Note that the TTL
357    * is so short that newly allocated objects might actually expire before a
358    * claim call can complete, and thus more than the expected two allocations
359    * are possible. This is why we check for *at least* two allocations.
360    * @param fixture
361    * @throws Exception
362    */

364   @Theory public void
365   mustReplaceExpiredPoolables(PoolFixture fixture) throws Exception {
366     Pool pool = fixture.initPool(
367         config.setTTL(1, TimeUnit.MILLISECONDS));
368     pool.claim().release();
369     spinwait(2);
370     pool.claim().release();
371     assertThat(allocator.allocations(), greaterThanOrEqualTo(2));

374   /**
375    * The size limit on a pool is strict, unless specially (as in a
376    * non-standard way) configured otherwise. A pool is not allowed to 
377    * have more objects allocated than the size, under any circumstances.
378    * So, when the pool renews an object it must make ensure that the
379    * deallocation of the old object happens-before the allocation of the
380    * new object.
381    * We test for this by configuring a pool with a timeout of one millisecond.
382    * Then we claim an release an object, and wait for 2 milliseconds. Now the
383    * object is expired, and must therefore be re-allocated before the next
384    * claim can return. So we do a claim (but no release) and check that we have
385    * had at least one deallocation. Note that our TTL is so short, that an
386    * object might expire before a claim call can complete, so this is why we
387    * check for *at least* one deallocation.
388    * @param fixture
389    * @throws Exception
390    * @see Config#setSize(int)
391    */
392   @Test/*(timeout = 300)*/
393   @Theory public void
394   mustDeallocateExpiredPoolablesAndStayWithinSizeLimit(PoolFixture fixture)

397         config.setTTL(1, TimeUnit.MILLISECONDS));
398     pool.claim().release();
399     spinwait(2);
400     pool.claim();

404   /**
405    * When we call shutdown() on a pool, the shutdown process is initiated and
406    * the call returns a Completion object. A call to await() on this
407    * Completion object will not return until the shutdown process has been
408    * completed.
409    * A shutdown process is not complete until all Poolables in the pool have
410    * been deallocated. This means that any claimed objects must be released,
411    * all the deallocations must have returned.
412    * We test for this effect by making a pool of size 2 and claim both objects.
413    * Then we release them. The order is important, to prevent the allocation
414    * of just one object that is then reused. Then we shut the pool down and
415    * wait for it to finish. After this, we must observe that exactly 2
416    * deallocations have occurred.
417    * @param fixture
418    * @throws Exception
419    */
420   @Test(timeout = 300)
421   @Theory public void
422   mustDeallocateAllPoolablesBeforeShutdownTaskReturns(PoolFixture fixture)
423   throws Exception {
424     Pool pool = fixture.initPool(config.setSize(2));
425     Poolable p1 = pool.claim();
426     Poolable p2 = pool.claim();
427     p1.release();
428     p2.release();
429     shutdown(pool).await();
430     assertThat(allocator.deallocations(), is(2));
431   }

581    * To test this, we create a pool with one object and claim it. Then we

631    * a user error, but pools must still maintain a coherent allocation and
632    * deallocation pattern toward the Allocator.
633    * We test this by configuring a pool with a short TTL so that the objects
